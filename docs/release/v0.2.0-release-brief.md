# Steno v0.2.0 Release Brief

## Release Summary

Steno v0.2.0 is the first release that fully combines the redesigned macOS product surface with the recent engine-quality and release-eval work for this release cycle. Compared to public `v0.1.10`, this release ships a full shell redesign across Record, History, Settings, and onboarding; richer local transcription metadata and hardware/model guidance; stronger cleanup, no-speech, and repair handling; and a formal release-eval workflow with machine-readable signoff artifacts. The current canonical 0.2 candidate validates the exact `m5-pro / 64GB / large-v3-turbo` row on the 2026-04-23 release-signoff run.

## Redesigned Product Surface

- **App shell:** The window now uses a custom macOS surface with its own title-bar treatment, navigation, atmosphere controls, and a more intentional visual hierarchy.
- **Record:** The old simple recorder has been replaced by a more expressive hero layout with distinct idle/recording/transcribing states, compact metrics, richer transcript presentation, and inline cancel affordances.
- **History:** Transcript browsing is now a real workspace with clearer grouping, stronger row states, preview behavior, and better inline actions.
- **Settings:** Settings now read like a product surface instead of a form dump, including appearance controls, model guidance, and cleaner permission/insertion layouts.
- **Onboarding:** First-run setup now matches the redesigned visual system and better explains the local-first workflow.
- **Overlay:** The floating macOS dictation overlay now uses waveform motion, accent-aware styling, terminal-state icons, and compact cancel behavior.
- **App icon:** The shipped macOS icon now uses the new app logo instead of the older blue glyph.

## Engine and Cleanup Improvements

- Whisper transcription now prefers rich JSON output when available and preserves timing/confidence metadata instead of flattening everything to plain text too early.
- Prompt steering is now assembled from language hints, app metadata, and hot terms, which improves recovery for product names and app-specific terms without changing the local-only model.
- `SessionCoordinator` now short-circuits prompt-echo-like and empty transcripts to `.noSpeech` before cleanup, insertion, or history persistence.
- Cleanup remains local-first, but the candidate generation and ranking path is now more repair-aware and more robust to punctuation variance in natural speech.
- The repair matcher is shared between candidate generation and ranking to avoid drift, and repair-join cleanup strips punctuation around the repair boundary more reliably.
- Command passthrough safety remains conservative; the current release corpus still does not provide an honest evaluated raw-leading-slash command sample.

### Important interpretation rules

- Smoke fixtures prove the repo-level validation path works; they are not release evidence.
- The exact validated row is row-specific evidence, not a claim that all Pro or Max Macs are validated.
- WER and CER are lower-is-better error rates. Public copy should show both the before/after rates and the relative error reduction, not a naked “accuracy improved” claim.
- `commandPassthroughAccuracy` remains `not_evaluable` because the raw benchmark pass did not preserve a leading slash in the command corpus sample.
- Manual macOS sanity remains tracked separately from the blocking metric gate.

## Compatibility Matrix Status

- The compatibility matrix already marks `m5-pro / 64GB / large-v3-turbo` as `validated`.
- Other matrix rows remain `allowed-warning` unless they have their own exact release-signoff evidence.
- README/Quickstart copy should describe other rows as recommended or conservative defaults, not as release-validated.
- Any public wording about “best model for your Mac” must stay consistent with the matrix notes in `StenoKit/Sources/StenoKit/Resources/whisper-compatibility-matrix.json`.

## Known Limitations and Honest Caveats

- Release-eval validation is specific to the exact `m5-pro / 64GB / large-v3-turbo` row and should not be generalized to all Apple silicon hardware.
- The canonical corpus includes LibriSpeech plus synthetic targeted speech/noise/silence samples. That is useful release evidence, but it is not the same thing as universal production microphone validation.
- Manual macOS sanity is still marked `pending` in the current canonical report and should be treated as a separate final verification surface.
- Command passthrough still needs a corpus path that preserves the raw leading slash through the raw benchmark pass before that metric can move from `not_evaluable` to truly evaluated.
- Repair cleanup is materially stronger than before, but it is not correct to describe it as perfect. The current release corpus still reports `58.33%` repair-marker preservation in raw pipeline output even though trigger detection and cleanup outcomes now pass the release gates.
